## Основы

```python
from unittest.mock import Mock, MagicMock

mock = Mock()

mock.some_not_exist_method()
```

У Mock есть магический метод `__getattr__`, который создаёт моки для вызываемых методов на лету.
Потому нам даже не обязательно его объявлять. Достаточно просто вызвать, всё произойдёт автоматически.

```python
result = mock()
result = mock.any_method()
result = mock.any_attribute
result = mock.any_attribute.and_more.and_one_more
```

На каждый вызов мы получаем новый мок.

## Присваивание значений

Атрибутом мока может быть что угодно: число, строка, словарь, лист, функция, другой мок и т.д.
Значение рисваивают атрибутам стандартным способом через `=`:

```python
mock.some_attribute = "some_value"
```

Под капотом действует дандер `__setattr__`. 
Такая запись эквивалентна `setattr(normal_mock_example, "some_attribute", "some_value")`.

Если у нашего мока есть функция, её результат можно замокать с помощью `result_value`.

```python
mock.somemethod.result_value = False
```

Если мы хотим чтобы у нашего мока был аргумент в виде функции и потом хотим эту функцию вызвать, мы можем поступить так:

```python
def funk(n):
    print(n * 2)


mock.attr = funk
mock.attr(5) # Вернёт результат выполнения функции.
```

Однако тут есть загвоздка. 
О ней ниже, в главе `side_effect`.


## Атрибуты моков

```python
mock.called  # Вернёт True, если мок вызывали хотя бы один раз. Если не вызывали - False.
mock.call_count  # Вернёт количество вызовов мока числом.

mock("arg_1", {"dict_agr_2": "some_value"})

mock.call_args  # Вернёт специальный объект с набором аргументов, с которым вызывали мок в последний раз.
# Пример:
# call("arg_1", {"dict_agr_2": "some_value"})

mock(3, [0, 1, 2])
mock.call_args  # Вернёт уже другой набор аргументов - тех, что были переданы в последний раз.


mock.call_args_list  # Вернёт список список спец.объектов с аргументами.
# Это уже настоящая история вызовов.
# Пример:
# [
#     call("Alice"),
#     call("Bob"),
#     call("Charlie")
# ]

mock.mock_calls  # Вернёт список всех вызовов. Не только самого мока, но и его аргументов.
# Пример:
# [
#     call.login("Alex"),
#     call.send_message("Привет"),
#     call.logout()
# ]
```

## Методы assert_*

В тестах при проверках чаще используют методы `assert_*`, а не проверки по типу:

```python
assert mock.call_count == 1
```

Эти методы отдают более понятную ошибку, если что-то пошло не так.
Пример:

```python
Expected call: ("Alex")
Actual call: ("Kate")
```

#### 1 группа: Вызывались ли вообще?

```python
mock.assert_called()   Мок был вызван хотя бы один раз.
mock.assert_called_once()   Мок был вызван не больше и не меньше, чем один раз.
mock.assert_not_called()   Мок не вызывался.
```

#### 2 группа: С какими аргументами вызывались?

```python
mock.assert_called_with()   В последний раз мок был вызван именно с этими аргументами.
mock.assert_called_once_with()   Мок был вызван всего 1 раз и именно с этими аргументами.
mock.assert_any_call()   Хотя бы раз мок вызывался именно с этими аргументами.
```

#### 3 группа: В каком порядке происходили вызовы?

```python
mock.assert_has_calls()  # Цепочка вызово соответствует той, что мы ожидаем.
# Пример запроса:

from unittest.mock import call

mock.assert_has_calls(
    [
        call.login(),
        call.send("Hello"),
        call.logout(),
    ]
)

# По умолчанию - проверяет порядок вызовов.
# Если он нам не важен, мы можем указать параметр any_order = True.

mock.assert_has_calls(
    [
        call.login(),
        call.send("Hello"),
        call.logout(),
    ],
    any_order=True,
)
```

## side_effect

#### Задать фиксированное поведение
___

`return_value` - что вернуть при вызове.
`side_effect` - что сделать при вызове.

`side_effect` позволяет настроить поведение при вызове мока.
Например, если нам надо чтобы при каждом вызове нам отдавало разный результат, мы можем сделать так:

```python
# Мок "Часы"

mock.side_effect = [
    "10:00",
    "10:01",
    "10:02"
]
```

При каждом новом вызове мы получим новый, заранее заготовленный ответ.

В данном примере мы заложили 3 вызова. На 4-й мы получим исключение `StopIteration`.
Потому что на самом деле список внутри `side_effect` превращается в итератор.
А раз список закончился, дальнейшие итерации не возможны.

#### Генерация исключений
___

Так же `side_effect` позволяет нам генерировать исключения.
Например, когда мы ожидаем, что вызов нашего мока должен приводить к ошибке, мы можем заложить такое поведение.

```python
mock.side_effect = TimeoutError
# или
mock.side_effect = TimeoutError("Сервер не отвечает")
```

#### Работа с функциями
____

Ещё мы можем передать side_effect настоящую функцию.

Выше, в Основах мы выяснили, что атрибуту мока можно присвоить функцию.
Однако в таком случае `mock.attr`, которому присвоена функция, перестаёт быть моком.
Он заменён обычной функцией и потому все "плюшки" мокирования перестают быть доступными.

Например методы `assetr_*` больше работать не будут, потому что функция так не умеет.

`side_effect` действует иначе.
Тут мы не заменям мок функцией. Мок сам вызывает эту функцию и возвращает её результат.

```python
def greet(name):
    return f"Привет, {name}!"

mock = Mock()
mock.side_effect = greet

mock("Анна") # Выведет результат функции "Привет, Анна!"
```

* - работает и с несколькими аргументами.
Всё передаст, ничего не потеряет по пути.

При этом он продолжает вести учёт вызовов.
Все аргументы мока продолжают работать.

Мы так же сможем проверить и количество вызовов и историю аргументов.

> А что если я укажу и `return_value` и `side_effect` одновременно?

```python
mock = Mock()

mock.return_value = 100

mock.side_effect = lambda x: x * 2
mock(5)
```

`side_effect` выиграет.
Может быть не очевидно. Я сама подумала, что раз мы задаём условие "Отдавай всегда 100", значит у return_value будет преимущество.
Ведь всегда - это в том числе и "в результате выполнения функции".

Но разработчики моков решили иначе: ***Живое проведение приоритетнее заглушки***

## patch

Когда мы импортируем существующий объект из другого файла, мы храним его в локальной ссылке.
`patch` позволяет подменить эту ссылку на фейковую.

Потому очень важно патчить именно в том файле, в который мы импортируем объект, а не в файле самого объекта.

Условно:

```python
# Файл service.py
def send_email(user):
    print("Email sent to", user)
```

```python
# Файл app.py
from service import send_email

def notify():
    send_email("Anna")
```

Неправильно:
`patch("service.send_email")`

Правильно:
`patch("app.send_email")`

Когда в своём тесте мы пишем:

```python
from unittest.mock import patch

with patch("app.send_email") as mock_send:
    notify()

mock_send.assert_called_once_with("Anna")
```

Python находит `app.send_email`, сохраняет оригинал, подменяет на мок.
А после вызова возвращает обратно.

Т.е: изначально у нас была ссылка на функцию, переменная в которую она сохранена.
Мы в этой переменной при вызове вместо оригинальной функции вставили мок. А после - вернули как было.

У patch есть 3 формы:

1. Контекстный менеджер

```python
with patch("app.send_email") as mock:
    notify()
```

2. Декоратор

```python
@patch("app.send_email")
def test_notify(mock_send):
    notify()
```

3. Ручной

```python
p = patch("app.send_email")
mock = p.start()

notify()

p.stop()
```
